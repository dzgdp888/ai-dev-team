# ai-dev-team

> 基于 Claude Code 多 Agent 协作的全流程软件研发自动化框架——从需求到交付，一句话启动完整研发团队。

**ai-dev-team** 把一个完整的软件研发团队封装成一套可复用的框架模板。你只需描述要做什么，说一句"启动开发团队"，框架会自动协调 11 个专职 Agent 分工协作，经历需求分析、技术架构、前后端并行开发、代码审核、三轮测试、本地部署的全流程，最终交付一个可以 `docker-compose up` 跑起来的完整项目。

整个过程中，你只需要在 4 个关键节点做决策（确认 PRD、确认技术方案、确认视觉风格、验收交付），其余环节由 Agent 流水线自主推进。每个 Agent 只负责自己的职责范围，不越权、不代劳，通过严格的"审核 → 测试 → 部署"质量闸门，确保交付质量。

**适合场景：**

- 需要快速落地一个完整 Web 应用原型
- 个人开发者想用 AI 扩展自己的研发能力边界
- 小团队希望有一套标准化的研发流程

**默认技术栈：** Vue 3 + Element Plus · Python + FastAPI · MySQL · Docker Compose

> 当前版本：**V1**，框架持续迭代中——后续将陆续扩展更多 Agent 角色、支持更多技术栈，以及优化多里程碑大型项目的协作流程。

## 框架简介

本框架将软件研发流程拆解为多个专职 Agent，由 Claude Code 主会话作为**透明基础设施层**统一调度。每个 Agent 只负责自身职责范围内的工作，通过严格的流水线和用户确认节点，实现高质量的全流程软件交付。

**核心设计理念：**

- **分工明确**：每个 Agent 只做自己的事，不越权、不代劳
- **流程可控**：4 个关键节点等待用户确认，用户全程掌握主动权
- **并行提效**：前后端开发同时推进，不串行等待
- **质量闭环**：代码审核 → 接口测试 → UI 测试 → 集成测试 → 部署验证，全流程质量把关

**框架文件结构：**

```
ai-dev-team/          ← 本目录（框架模板）
├── README.md          ← 本说明文档
├── CLAUDE.md          ← 主会话行为规范（需复制到项目根目录）
├── AGENTS.md          ← 团队配置、技术栈、工作流、全局规则（需复制到项目根目录）
├── agents/            ← 各 Agent 定义文件（需复制到 ~/.claude/agents/）
└── local-images.txt   ← 本地可用的 Docker 镜像版本（需复制到项目根目录，可选）
```

## 快速开始

### 前置要求

- [Claude Code](https://claude.ai/code) CLI 已安装并登录
- Docker 20.10+ 和 docker-compose 2.0+

### 安装

**第一步：** 将 `agents/` 目录下的所有 `.md` 文件复制到 Claude Code 的全局 agents 目录：

```bash
cp agents/* ~/.claude/agents/
```

**第二步：** 将以下文件复制到你的**项目根目录**：

```bash
cp CLAUDE.md /path/to/your/project/
cp AGENTS.md /path/to/your/project/
# 如需固定镜像版本，也一并复制：
cp local-images.txt /path/to/your/project/
```

> `AGENTS.md` 定义了完整的技术栈约束、团队规则和工作流，是框架运行的核心配置，**必须**放在项目根目录。`local-images.txt` 用于指定本地可用的 Docker 镜像版本，若无离线/版本限制需求可不复制。

### 启动一个新项目

在项目目录下打开 Claude Code，然后：

```
需求：[描述你要做的系统，尽量详细]
启动开发团队。
```

项目经理将接管后续所有协调与沟通，你只需在 4 个确认节点做决策。

### 本地镜像版本控制（可选）

如果你的环境需要使用特定版本的 Docker 镜像，在项目根目录创建 `local-images.txt`：

```
mysql:8.0.36
redis:7.2.4
python:3.11-slim
nginx:1.25
elasticsearch:8.11.0
```

本地部署工程师会优先使用此文件中的版本，不自行升级或替换。

## 完整工作流程

```
用户输入需求 + "启动开发团队"
    ↓
① 项目经理 → 接收需求，评估可行性，启动流程
    ↓
② 产品经理 → 需求分析，输出 docs/prd.md
    ↓
【确认节点一：用户确认 PRD】← 等待用户回复"确认"
    ↓
③ 项目经理 → 基于 PRD 拆解任务，制定里程碑
    ↓
④ 技术架构师 → 系统设计，输出 docs/architecture.md + docs/api-spec.md
    ↓
【确认节点二：用户确认技术方案】← 等待用户回复"确认"
    ↓
【确认节点二点五：用户确认前端视觉风格】← 等待用户回复（或"默认"）
    ↓
⑤ 项目经理 → 输出 docs/milestone-1.md（本里程碑范围）
    ↓
⑥ 后端开发 ─────────────────────────── 同时启动，并行执行
   前端开发（基于 api-spec.md + Mock）──────────────────────
    ↓
⑦ 代码审核 → 输出 docs/review-result.md
    ├─ 有阻塞问题 → 返回对应开发修复 → 重新审核（最多3轮）
    └─ 通过 → 进入测试
    ↓
⑧ 接口测试（后端 API）─── 并行执行
   UI 测试（前端页面）────────────────
    ├─ 不通过 → 返回对应开发修复 → 重新测试
    └─ 两项均通过 → 集成测试
    ↓
⑨ 集成测试（E2E 完整流程）
    ├─ NEEDS WORK → 返回对应开发修复 → 重新集成测试
    └─ PASS → 继续
    ↓
⑩ 本地部署工程师 → 编写/检查 Dockerfile + docker-compose.yml + .gitignore → 启动服务
    ├─ 启动失败 → 排查修复 → 重新启动
    └─ 启动成功 → 输出 docs/deploy-report.md
    ↓
【确认节点三：用户验收里程碑 N】← 等待用户回复"确认"
    ├─ 有后续里程碑 → 项目经理输出 milestone-N+1.md → 返回步骤⑥
    └─ 全部完成 → 继续
    ↓
⑪ 技术文档工程师 → 输出 README.md 及技术文档
    ↓
（可选）售前顾问 + 文档工程师 → 输出售前 PPT/PDF
    ↓
项目经理 → 宣布交付完成
```

### 并行开发说明

前后端进入开发阶段时**必须同时启动**，不得串行等待。

- **后端**：依据 `docs/milestone-N.md` + `docs/api-spec.md` 实现接口
- **前端**：依据 `docs/milestone-N.md` + `docs/api-spec.md` + Mock 数据开发，无需等待后端完成

`docs/api-spec.md` 是前后端并行的**唯一契约文件**，由技术架构师在技术方案阶段一并输出。

## 用户确认节点

框架设计了 4 个硬性暂停点，到达后所有 Agent 必须停止，等待用户明确回复"确认"或"继续"。

### 节点一：PRD 确认

产品经理完成 PRD 后触发。项目经理输出：

- 文档路径：`docs/prd.md`
- 系统核心功能列表
- 明确不包含的功能
- 核心用户角色
- 验收标准摘要

> 用户可直接回复"确认"，或说明修改意见。

### 节点二：技术方案确认

技术架构师完成方案后触发。项目经理输出：

- 文档路径：`docs/architecture.md`
- 本项目启用/未启用的中间件及理由
- 系统模块划分
- 关键技术决策
- 接口设计摘要

> 用户可直接回复"确认"，或说明修改意见。

### 节点二点五：前端视觉风格确认

技术方案确认后、前端开发启动前触发。项目经理询问：

- 是否有参考风格（截图路径或网址）
- 主色调偏好
- 整体感觉（商务感 / 科技深色感 / 清新轻量感 / 数据仪表盘风格）

> 无特殊要求直接回复"默认"，前端使用 Element Plus 标准风格。

### 节点三：里程碑交付确认

每个里程碑全部任务完成、审核和测试均通过后触发。项目经理输出：

- 本里程碑完成的功能列表
- 测试结果（通过/总用例数）
- 可访问地址（前端、后端 API 文档）
- 下一里程碑计划

> 如发现问题直接描述，项目经理会定位并唤起对应 Agent 修复。

### 节点四：重大变更确认（随时触发）

任何涉及需求范围、技术方案、里程碑计划的变更时触发，项目经理输出变更内容、原因、影响范围和建议方案，等待用户决策。

## 目录结构说明

框架运行期间，项目目录结构如下：

```
project/
├── CLAUDE.md                  ← 主会话行为规范
├── AGENTS.md                  ← 团队配置（从 ai-dev-team/ 目录复制）
├── agents/                    ← Agent 定义文件（从 ai-dev-team/ 目录复制）
├── .gitignore                 ← 本地部署工程师生成
├── local-images.txt           ← 本地 Docker 镜像版本（可选）
├── requirements.md            ← 用户需求（开始前填写，可选）
├── status.md                  ← 实时进度（各 Agent 自动写入）
├── docker-compose.yml         ← 本地一键启动
├── docs/                      ← 所有文档产出
│   ├── prd.md                 ← 产品需求文档
│   ├── architecture.md        ← 架构设计文档
│   ├── api-spec.md            ← 接口契约（前后端并行开发的依据）
│   ├── ui-spec.md             ← 前端视觉规范
│   ├── ui-reference/          ← 用户提供的参考截图
│   ├── milestone-1.md         ← 里程碑 1 任务说明
│   ├── review-result.md       ← 代码审核结果
│   └── deploy-report.md       ← 部署报告
├── frontend/                  ← Vue 3 前端代码
│   ├── Dockerfile
│   ├── nginx.conf
│   └── src/
└── backend/                   ← FastAPI 后端代码
    ├── Dockerfile
    ├── .env
    └── app/
```

### `status.md` 格式

每个 Agent 完成关键步骤后，自动追加写入：

```
## [YYYY-MM-DD HH:MM] [角色名]
**状态**: 进行中 / 已完成 / 等待确认 / 阻塞
**当前步骤**: 正在做什么
**新增文件**: 本次新创建的文件
**修改文件**: 本次改动的已有文件
**下一步**: 交给谁做什么
**备注**: 风险、依赖等（无则省略）
```

`status.md` 顶部还维护一份任务清单（`- [ ]` / `- [x]`），全团队实时同步更新。

## 核心配置文件

### CLAUDE.md — 主会话行为规范

每次新会话时自动加载，定义主会话的核心职责：

- 读取 `AGENTS.md` 加载配置
- 按流水线顺序唤起对应 Agent
- 将 Agent 产出原样呈现给用户
- 前端视觉风格确认后，负责将参考截图复制到 `docs/ui-reference/`
- **不输出自己的分析、建议或总结**
- **不替代任何 Agent 的职责**
- **不直接修复 bug**

### AGENTS.md — 团队总配置

完整定义了：

- 团队成员构成和职责边界
- **强制技术栈和禁止技术**（前端、后端、数据库、中间件、大模型、容器化、接口规范）——如需调整技术选型，直接修改此文件
- 完整工作流程图
- 4 个确认节点的精确格式模板
- 全局执行规则（进度追踪、质量规范、Bug 处理、代码审核、README 规范等）
- 里程碑划分原则
- 各 Agent 的严禁行为清单

### local-images.txt — 本地镜像版本

项目根目录的可选文件，格式为每行一个镜像全名（含版本标签）：

```
mysql:8.0
redis:7-alpine
python:3.11-slim
nginx:1.25
elasticsearch:8.11.0
```

优先级高于 Agent 默认版本，确保离线或受限环境下使用正确的镜像。

## 常见场景指南

### 场景一：全新项目开发

```
需求：开发一个企业内部知识库系统，支持文档上传、全文检索和 AI 问答。
启动开发团队。
```

框架将自动完成需求分析 → 技术方案 → 并行开发 → 测试 → 部署的全流程。

### 场景二：报告 Bug

在任何阶段直接描述问题，例如：

```
登录接口返回 500 错误，错误信息是：...
```

项目经理会自动判断归属（前端/后端），唤起对应开发 Agent 修复，经审核和测试后汇报结果。

**注意：** 主会话和项目经理都不会自行修复 bug，必须唤起专职开发 Agent。

### 场景三：需求变更

直接告知变更内容：

```
需要增加微信扫码登录功能。
```

项目经理会评估影响范围，触发"重大变更确认"节点，等待你确认后重置受影响的任务状态并重新安排。

### 场景四：注入演示数据

系统部署完成后：

```
请为演示环境注入测试数据。
```

框架会激活测试数据生成工程师，通过调用已部署的 API 注入符合业务逻辑的演示数据。

### 场景五：输出售前材料

系统验收后：

```
请基于本系统输出一份售前方案 PPT。
```

框架会激活售前顾问 + 文档工程师，输出 `.pptx` / `.pdf` 格式的售前材料。

### 场景六：多里程碑大型项目

对于功能模块较多的项目（超过 3 个主要模块），项目经理会自动拆分里程碑：

- 每个里程碑结束时都有"用户验收确认"节点
- 用户确认后才启动下一里程碑的开发
- `status.md` 任务清单实时反映当前里程碑进度

## 注意事项

### Agent 职责边界

| 禁止行为 | 原因 |
|---------|------|
| 项目经理编写任何业务代码 | 职责越权，破坏分工 |
| 主会话直接修复 bug | 绕过质量流程 |
| 跳过代码审核直接测试 | 质量闸门不可跳过 |
| 前后端串行开发 | 降低效率 |
| 开发 Agent 实现里程碑范围外的功能 | 防止需求蔓延 |

### 代码审核流程

审核结果写入 `docs/review-result.md`，分三级：

- 🔴 **阻塞问题**：必须修复后才能进入测试，最多循环 3 轮
- 🟡 **建议优化**：不阻塞，可后续处理
- 💭 **Nit**：可忽略的细节

### 里程碑划分原则

- **默认一个里程碑完成全部开发**，能一次做完的不拆
- 只有功能模块数量多、预估超过 3 个主要模块时，才拆分为多个里程碑

### 技术文档要求

最终交付时，技术文档工程师输出的 `README.md` 必须包含：项目描述、技术栈、前置要求、快速启动（三步）、访问地址、环境变量说明、项目结构、主要功能、开发指南、常用命令。

## 关于 agent 框架来源

本目录中的 Agent 文件基于 [agency-agents](https://github.com/msitarzewski/agency-agents) 开源项目定制，针对以下方面进行了本地化适配：

- 统一技术栈（Vue 3 + FastAPI + MySQL + docker-compose）
- 中文化团队协作规范
- 新增项目经理严禁行为清单
- 新增前端视觉风格确认节点
- 新增 `local-images.txt` 本地镜像版本控制机制
- 新增测试数据生成工程师等可选角色
