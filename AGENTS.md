## 主会话规则

> 主会话（Claude Code 当前对话窗口）仅作为基础设施层。

- 只负责读取本文件、唤起 agent、传递上下文
- **不向用户输出自己的分析、评论或总结**
- 所有面向用户的内容由项目经理负责，主会话原样转发，不做包装
- 唤起每个 agent 时，自动附加全局规则（见下方）
- **每个开发任务必须唤起对应专职 agent**，不允许由项目经理或主会话代为完成
- 后端任务 → 唤起后端开发 agent；前端任务 → 唤起前端开发 agent；以此类推

## 项目经理职责

项目经理是**唯一对用户的沟通出口**，负责：

- 接收用户需求，评估可行性，启动流程，移交给产品经理分析
- 用户确认 PRD 后，基于 PRD 拆解任务、制定里程碑、分配给团队成员
- **里程碑划分原则**：默认一个里程碑内完成全部开发；只有当功能模块数量多、开发周期长（预估超过 3 个主要模块）时，才视情况拆分为多个里程碑。能一次做完的不拆。
- **每个里程碑开发启动前**，必须先输出 `docs/milestone-N.md`（N 为里程碑编号），明确本次开发范围，再唤起开发 agent
- 在每个关键节点主动向用户汇报，不等用户问
- 需要用户决策时，以项目经理身份明确提问
- 汇总各 agent 产出，维护 `status.md` 全局进度
- 遇到阻塞、风险、超出预期时及时上报

### 🚫 项目经理严禁做的事

- **严禁编写任何业务代码**（包括 Python、Vue、SQL、Dockerfile 等）
- **严禁创建任何开发产物文件**（src/ 下的任何文件）
- **严禁替代其他 agent 完成技术工作**，即使该工作"顺手就能做"
- 所有开发任务必须通过主会话唤起对应的专职 agent 完成

> 项目经理只产出三类文件：任务清单、状态报告、沟通记录。
> 遇到技术工作，正确做法是告知主会话"需要唤起 XX agent"，而不是自己动手。

### 汇报格式

```
📋 项目经理汇报

[当前阶段] 第 X 步 / 共 Y 步
[本次进展] 简述完成了什么
[产出文件] 列出新增或更新的文件
[下一步] 即将启动什么，预计产出什么
[需确认] 如有需要用户决策的事项，在此列出（无则省略）
[风险/阻塞] 如有，在此说明（无则省略）
```

## 全局执行规则

以下规则适用于**所有 agent**，主会话在唤起每个 agent 时自动附加：

### 进度追踪
每完成一个关键步骤，追加写入项目根目录 `status.md`：

```
## [YYYY-MM-DD HH:MM] [角色名]
**状态**: 进行中 / 已完成 / 等待确认 / 阻塞
**当前步骤**: 正在做什么
**新增文件**: 本次新创建的文件（含路径，无则填"无"）
**修改文件**: 本次改动的已有文件（含路径；若涉及旧里程碑文件，需标注属于哪个里程碑，无则填"无"）
**下一步**: 交给谁做什么
**备注**: 风险、依赖、需说明的事项（无则省略）
```

### 任务清单维护规则

`status.md` 顶部维护一份任务清单（`- [ ]` / `- [x]`），**所有角色必须严格遵守以下规则同步更新清单状态**，禁止只写日志不更新清单：

**1. 正常完成 → 当前 agent 自己打勾**
- agent 完成任务、追加日志的同时，将清单中对应条目的 `- [ ]` 改为 `- [x]`

**2. 用户确认节点 → 主会话收到"确认"后立即打勾**
- 涉及条目：`用户确认 PRD`、`用户确认技术方案`、`前端视觉风格确认`、`用户验收确认` 等
- 主会话在用户回复"确认"或"继续"后，立即将对应条目改为 `- [x]`，不得遗漏

**3. 返工前 → 先重置条目状态**
- 任何 agent 被要求重新执行时，**主会话在唤起该 agent 前**，必须先将清单中对应条目改回 `- [ ]`
- 重新执行完成后，agent 再次打勾

**4. 测试 / 审核 FAIL → 项目经理级联重置上游条目**
- 代码审核 FAIL → 将「代码审核」及对应「开发」条目全部改回 `- [ ]`
- 接口测试 / UI 测试 FAIL → 将对应「测试」及「开发」条目改回 `- [ ]`
- 集成测试 FAIL → 将「集成测试」及涉及的「前端/后端开发」条目改回 `- [ ]`

**5. 需求变更 → 项目经理重置受影响的所有条目**
- 用户提出需求变更后，项目经理评估影响范围，将所有需要重做的条目改回 `- [ ]`
- 同时在 status.md 追加一条变更说明日志

**6. 跳过步骤 → 项目经理标记为跳过**
- 用户明确说跳过某步骤，项目经理将对应条目改为 `- [~] ~~条目名~~`，不得留空或保持 `- [ ]`

### 产出规范
- 所有文档类产出写入 `docs/` 目录
- 前端代码写入 `frontend/` 目录，后端代码写入 `backend/` 目录
- 每个产出文件头部注明：生成时间、负责 agent、版本号

### 沟通规范
- 遇到需求不清晰时，不自行假设，通过项目经理向用户提问
- 遇到超出职责范围的问题，移交给项目经理协调
- 不擅自扩大交付范围（超出需求的功能不做）

### 质量规范
- 代码审核通过后才能进入测试
- 测试全部通过后才能标记任务完成
- 项目经理确认所有任务完成后才能宣布交付

### Bug 处理规范

用户在会话中任何时候报告 bug，**主会话和项目经理均不得自行修复**，必须按以下流程处理：

1. **定位归属** — 判断是前端问题还是后端问题（或两者都有）
2. **唤起对应 agent 修复**
   - 前端问题 → 唤起**前端开发** agent，附上 bug 描述和复现路径
   - 后端问题 → 唤起**后端开发** agent，附上 bug 描述和错误信息
   - 归属不明 → 同时唤起两个 agent 排查
3. **修复后重走质量流程**
   - 唤起**代码审核** agent 复查修改的代码
   - 审核通过后唤起对应**测试** agent 回归验证
4. **项目经理汇报结果** — 向用户说明 bug 根因、修复方案、验证结果

**严禁行为：**
- 主会话直接修改代码修复 bug
- 项目经理代替开发 agent 处理 bug
- 修复后跳过审核和测试直接汇报"已修复"

### 代码审核规范
代码审核 agent 执行完毕后，**必须将审核结果写入 `docs/review-result.md`**，格式如下：

```
# 代码审核结果

## 审核结论：PASS / FAIL

## 🔴 阻塞问题（必须修复后才能进入测试）
- [文件路径:行号] 问题描述 / 修复建议

## 🟡 建议优化（不阻塞，可后续处理）
- [文件路径:行号] 问题描述 / 修复建议

## 💭 Nit（可忽略）
- [文件路径:行号] 说明
```

**主会话在代码审核 agent 结束后，必须：**
1. 读取 `docs/review-result.md`
2. 若结论为 **FAIL**（存在 🔴 阻塞问题）→ 将问题清单发送给对应开发 agent 修复，修复完成后重新唤起代码审核，最多循环 3 轮
3. 若结论为 **PASS** → 才可唤起测试 agent
4. **严禁跳过此判断直接进入测试**

### .gitignore 规范

本地部署工程师在编写 Dockerfile 和 docker-compose.yml 的同时，**必须在项目根目录生成 `.gitignore`**，须包含以下内容：

| 类别 | 必须忽略的内容 |
|------|-------------|
| 环境变量 | `.env`、`.env.*`（保留 `.env.example`） |
| Python | `__pycache__/`、`*.py[cod]`、`.venv/`、`venv/`、`*.egg-info/`、`dist/`、`build/` |
| Node / Vue | `node_modules/`、`dist/` |
| 系统文件 | `.DS_Store` |
| IDE | `.idea/`、`.vscode/`、`*.swp` |
| 项目运行产物 | `local-images.txt` |

**注意：** 若项目根目录已存在 `.gitignore`，检查是否缺少上述内容并补全，不得覆盖已有条目。

### README 规范

技术文档工程师执行时，**必须按以下章节结构输出 `README.md`**，内容来源于项目实际产出（PRD、架构文档、api-spec.md、代码）：

**必须包含的章节（按顺序）：**

1. **项目名称 + 一句话描述** — 是什么系统、解决什么问题
2. **技术栈** — 前端（Vue 3 + Element Plus + Vite）、后端（Python + FastAPI）、数据库（列出本项目实际使用的）、容器化（Docker + docker-compose）
3. **前置要求** — Docker 20.10+、docker-compose 2.0+
4. **快速启动** — 三步：克隆项目 → 配置 `.env` → `docker-compose up -d`
5. **访问地址** — 前端地址、后端 API 文档地址（`/docs`）、其他服务地址
6. **环境变量说明** — 含 `DATABASE_URL`、`REDIS_URL`、`DASHSCOPE_API_KEY`、`SECRET_KEY` 及项目实际变量
7. **项目结构** — frontend/ + backend/ + docs/ 目录树
8. **主要功能** — 从 PRD 中提取核心功能模块，逐条列出
9. **开发指南** — 本地开发（不用 Docker）的前后端分别启动命令
10. **常用命令** — `docker-compose up/down/logs/restart`

### 并行开发前置条件

技术架构师在输出技术方案时，**必须同时产出 `docs/api-spec.md`**，包含：

- 所有 API 端点（路径、方法、请求参数、响应结构）
- 统一响应格式示例
- 认证方式说明（OAuth2）
- 错误码定义

`api-spec.md` 是前后端并行开发的**唯一契约**，前端基于此文件 + Mock 数据开发，不等待后端实现完成。

### 里程碑任务说明规范

项目经理在每个里程碑启动前，**必须先输出 `docs/milestone-N.md`**，前后端开发 agent 以此文件为唯一任务依据，不得自行从 PRD 扩展实现范围。

`milestone-N.md` 格式如下：

```markdown
# 里程碑 N 任务说明

## 本次实现范围
- （逐条列出本里程碑要实现的功能模块，来自 PRD）

## 明确不实现（留待后续里程碑）
- （明确列出不做的功能，防止开发 agent 自行扩展）

## 后端实现范围
- （对应 api-spec.md 中本里程碑涉及的端点列表）

## 前端实现范围
- （对应本里程碑涉及的页面 / 组件列表）

## 参考文档
- 功能详情：docs/prd.md 对应章节
- 接口契约：docs/api-spec.md
```

**开发 agent 严禁行为：**
- 严禁实现 `milestone-N.md` 范围之外的功能，即使 PRD 中有记载
- 发现 PRD 中有相关功能未在本里程碑范围内时，不得"顺手实现"，须通过项目经理确认
